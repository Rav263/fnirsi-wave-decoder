# FNIRSI DPOX180H — Формат файлов .wav и структура прошивки

## Обзор

Осциллограф FNIRSI DPOX180H сохраняет осциллограммы в файлы с расширением `.wav`,
однако формат **не имеет ничего общего с RIFF/WAV** (аудио). Это проприетарный
бинарный формат с big-endian порядком байт, содержащий полный снимок состояния
осциллографа: настройки, скриншот экрана и сырые данные АЦП.

Первые байты файла — `00 00 00 32`, а не `52 49 46 46` (`RIFF`).

---

## Структура файла

Файл состоит из 4 секций, расположенных последовательно без промежутков:

```
Смещение          Секция                  Размер (типичный)
──────────────────────────────────────────────────────────
0x0000            Таблица секций          50 байт (0x32)
0x0032            Настройки               ~1019 байт
0x042D            Буфер экрана            ~11020 байт
0x2F39            Данные осциллограммы     ~6000 байт
                                          ─────────────
                  Итого:                   ~18089 байт
```

Все многобайтные значения — **big-endian**.

---

## 1. Таблица секций (0x00–0x31, 50 байт)

### 1.1. Смещения и размеры секций (0x00–0x1B)

Первые 28 байт (7 × u32 BE) описывают расположение трёх секций в файле:

| Смещение | Размер | Поле                    | Формат | Описание                              | Пример  |
|----------|--------|-------------------------|--------|---------------------------------------|---------|
| 0x00     | 4      | `settings_start`        | u32 BE | Начало секции настроек (всегда 50)    | 50      |
| 0x04     | 4      | `settings_size`         | u32 BE | Размер секции настроек                | 1019    |
| 0x08     | 4      | `screen_buffer_start`   | u32 BE | Начало буфера экрана                  | 1069    |
| 0x0C     | 4      | `screen_buffer_size`    | u32 BE | Размер буфера экрана                  | 11020   |
| 0x10     | 4      | `waveform_data_start`   | u32 BE | Начало данных осциллограммы           | 12089   |
| 0x14     | 4      | `waveform_data_size`    | u32 BE | Размер данных осциллограммы           | 6000    |
| 0x18     | 4      | `total_file_size`       | u32 BE | Полный размер файла                   | 18089   |

**Инвариант** — секции следуют строго друг за другом:
```
settings_start + settings_size     == screen_buffer_start
screen_buffer_start + screen_buffer_size == waveform_data_start
waveform_data_start + waveform_data_size == total_file_size
total_file_size == фактический размер файла
```

### 1.2. Зарезервированные байты (0x1C–0x31)

Оставшиеся 22 байта (0x1C–0x31) — назначение неизвестно. Во всех 20 тестовых
файлах значения идентичны:

```
0x1C: 00 00 02 FA   (u32 = 762)
0x20: F0 80 00 00   (u32 = 0xF0800000)
0x24: 00 00 01 7D   (u32 = 381)
0x28: 78 40 00 00   (u32 = 0x78400000)
0x2C: 00 00 00 98   (u32 = 152)
0x30: 96 80         (u16 = 0x9680)
```

Предположительно — параметры, зависящие от аппаратной конфигурации, но не от
захваченного сигнала.

| 0x2B6    | 2      | display_width                | 500 (константа)                   |

### 2.9. Таблица переменной длины (file 0x2B8+)

Сразу после поля `display_width` может располагаться таблица буферных границ
переменной длины. Количество записей — `var_table_count` (file 0x1E9).
Каждая запись — 3 × u32 BE (boundary_1, boundary_2, stride).

Все поля **после** этой таблицы сдвигаются на `var_table_count × 12` байт.

### 2.10. Вторая структура (DAT_0005d780)

Следует после таблицы переменной длины. Содержит:
- Количество отсчётов на канал / общее
- Конфигурацию измерений (частота, период и т.д.)

## 3. Буфер экрана (screen_buffer_start..+screen_buffer_size)

Миниатюра экрана осциллографа.

| Смещение (отн.) | Размер       | Поле     | Описание                     |
|------------------|--------------|----------|------------------------------|
| +0x00            | 2            | width    | Ширина в пикселях (102)      |
| +0x02            | 2            | height   | Высота в пикселях (54)       |
| +0x04            | width×height×2 | pixels | Данные RGB565 u16 BE         |

**Размер**: 4 + 102 × 54 × 2 = **11 020 байт**

Формат пикселей — предположительно RGB565 big-endian:
```
Бит 15..11 = Red   (5 бит)
Бит 10..5  = Green (6 бит)
Бит  4..0  = Blue  (5 бит)
```

---

## 4. Данные осциллограммы (waveform_data_start..+waveform_data_size)

Сырые 16-битные отсчёты АЦП (u16 BE), без сжатия.

### Расположение каналов

Каналы записываются последовательно — сначала CH1 (если включён), затем CH2:

```
if CH1_enabled (file[0x66] != 0):
    CH1_data = sample_count × u16 BE

if CH2_enabled (file[0xA6] != 0):
    CH2_data = sample_count × u16 BE
```

- `sample_count` = u32 BE по смещению **0x12D** (поле `buffer_sample_count`)
- `waveform_data_size` = `active_channels × sample_count × 2`

### Характеристики отсчётов

| Параметр           | Значение                      |
|--------------------|-------------------------------|
| Тип                | u16 big-endian                |
| Центр (0 В)        | 6400                          |
| Максимум АЦП       | 12800                         |
| Минимум АЦП        | 0                             |
| Кол-во (типичное)  | 3000 на канал                 |

### Пересчёт в напряжение

```
voltage_mV = (adc_value - 6400) × VDIV_TABLE[vdiv_index] / scale_factor
```

Где:
- `vdiv_index` — V/div index (file 0x6B для CH1, 0xAB для CH2)
- `scale_factor` — зависит от настроек вертикального масштаба

### Наблюдения по тестовым файлам

| Файл         | CH1 min | CH1 max | Диапазон | Клиппинг? |
|--------------|---------|---------|----------|-----------|
| 1MHz-1V.wav  | 5625    | 7155    | 1530     | Нет       |
| 1MHz-3V.wav  | 4162    | 8579    | 4417     | Нет       |
| 5MHz-1V.wav  | 3345    | 9482    | 6137     | Нет       |
| 5MHz-3V.wav  | 0       | 12800   | 12800    | Да        |
| 10MHz-1V.wav | 2866    | 10010   | 7144     | Нет       |
| 10MHz-3V.wav | 0       | 12800   | 12800    | Да        |

Сигналы 3V на высоких частотах (≥4 МГц) клиппируются по границам АЦП (0 и 12800).

## Структура прошивки (Ghidra-декомпиляция)

Прошивка декомпилирована из файла `DPOX180H-FNIRSI-V40.gzf` с помощью Ghidra.
Результат — файлы `.c` (декомпилированный C-код), `.h` (объявления) и `.lst`
(листинг дизассемблера).

### Ключевые функции сохранения WAV

| Функция          | Строка .c | Назначение                                        |
|------------------|-----------|---------------------------------------------------|
| `FUN_0005de70`   | 56619     | Обработчик UI «Сохранить» — управление файлами    |
| `FUN_0005da08`   | 56415     | Главная функция записи файла                      |
| `FUN_0005d878`   | 56348     | Координатор записи (вторичная точка входа)         |
| `FUN_0005c98c`   | 56003     | Сериализация состояния осциллографа (настройки)    |
| `FUN_000315d4`   | 9151      | Обёртка записи в файл `write(address, length)`    |
| `FUN_000317bc`   | —         | Flush / sync файла                                |
| `FUN_0003162c`   | —         | Seek / установка позиции в файле                  |
| `FUN_000318dc`   | —         | Получение текущей позиции в файле (ftell)          |

### Поток записи файла (FUN_0005da08)

```
1. FUN_000318c8()                     — открытие/подготовка файла
2. struct+0x838 = 0x32                — маркер: настройки начинаются с байта 50
3. FUN_0003162c()                     — seek к началу настроек
4. FUN_0005c98c()                     — запись секции настроек
5. checkpoint_1 = FUN_000318dc()      — запомнить текущую позицию
6. struct+0x844 = pos - struct+0x838  — вычислить размер настроек
7. struct+0x83c = pos                 — начало буфера экрана

                                      — ЗАПИСЬ БУФЕРА ЭКРАНА —
8. write(DAT_0005dba4, 2)            — ширина (u16)
9. write(DAT_0005dba4+1, 2)          — высота (u16)
10. loop: write(pixel[i], 2)          — width × height пикселей

11. checkpoint_2 = FUN_000318dc()
12. struct+0x848 = pos - prev_pos     — размер буфера экрана
13. struct+0x840 = pos                — начало данных осциллограммы

                                      — ЗАПИСЬ ДАННЫХ АЦП —
14. if CH1_enabled (struct+0x3c):
      for i in 0..sample_count:
        write(adc_buf[i*2], 2)        — u16 BE отсчёты CH1

15. if CH2_enabled (struct+0xec):
      for i in 0..sample_count:
        write(adc_buf[0x3B150+i*2], 2)— u16 BE отсчёты CH2

16. checkpoint_3 = FUN_000318dc()
17. struct+0x84c = pos - prev_pos     — размер данных
18. struct+0x850 = pos                — полный размер файла

                                      — ЗАПИСЬ ЗАГОЛОВКА (seek к 0x00) —
19. FUN_0003162c(0)                   — seek к началу файла
20. write(struct+0x838, 4)            — settings_start
21. write(struct+0x844, 4)            — settings_size
22. write(struct+0x83c, 4)            — screen_buffer_start
23. write(struct+0x848, 4)            — screen_buffer_size
24. write(struct+0x840, 4)            — waveform_data_start
25. write(struct+0x84c, 4)            — waveform_data_size
26. write(struct+0x850, 4)            — total_file_size
```

Заголовок записывается **последним** — прошивка сначала пишет данные, запоминает
позиции через ftell, а затем возвращается к началу файла и записывает таблицу секций.

### Ключевые глобальные переменные

| Адрес          | Тип           | Назначение                               |
|----------------|---------------|------------------------------------------|
| `DAT_0005dba0` | struct*       | Основная структура состояния осциллографа |
| `DAT_0005d780` | struct*       | Вторая структура (триггер/измерения)     |
| `DAT_0005dba4` | u16*          | Указатель на буфер экрана (W, H, pixels) |
| `DAT_0005dba8` | u16*          | Указатель на буфер данных АЦП            |
| `DAT_00031628` | file_context* | Контекст файлового дескриптора           |

| Смещение  | Тип    | Поле                   | Файл offset |
|-----------|--------|------------------------|-------------|

## Исправления dump_header.py

### 1. Переименование полей заголовка

**Было** (ошибочно):
```python
(0x00, 4, "header_const_1 (DAT_0005dbac)", "KNOWN", "firmware constant"),
(0x04, 4, "header_const_2 (DAT_0005dbb0)", "KNOWN", "firmware constant"),
(0x08, 4, "header_const_3 (DAT_0005dbb4)", "KNOWN", "firmware constant"),
(0x0C, 4, "header_const_4 (DAT_0005dbb8)", "KNOWN", "firmware constant"),
(0x10, 4, "data_start_offset",             "USED",  None),
(0x14, 4, "data_section_size",             "USED",  None),
```

Смещения 0x00–0x0C ошибочно считались «константами прошивки». В действительности
`DAT_0005dbac` — это **указатель** на `struct+0x838` в RAM (адрес 0x8006F620),
а функция `FUN_000315d4` записывает 4 байта **из** этого адреса. Это вычисляемые
смещения секций, а не фиксированные константы.

**Стало** (исправлено):
```python
(0x00, 4, "settings_start (struct+0x838)",      "USED", "always 50 (0x32)"),
(0x04, 4, "settings_size (struct+0x844)",       "USED", None),
(0x08, 4, "screen_buffer_start (struct+0x83c)", "USED", None),
(0x0C, 4, "screen_buffer_size (struct+0x848)",  "USED", None),
(0x10, 4, "waveform_data_start (struct+0x840)", "USED", None),
(0x14, 4, "waveform_data_size (struct+0x84c)",  "USED", None),
(0x18, 4, "total_file_size (struct+0x850)",     "USED", None),
```

### 2. Критическая ошибка чтения данных АЦП

**Было** (ошибочно):
```python
settings_size = u32(data, 0x10)       # 12089 — НЕ settings_size, а waveform_data_start!
data_section = u32(data, 0x14)        # 6000
adc_block = data_section // 2         # 3000
adc_start = settings_size + adc_block # 12089 + 3000 = 15089 — ПРОПУСКАЕТ первые 3000 байт!
ch_bytes = adc_block // 2             # 1500
spc = ch_bytes // 2                   # 750 отсчётов — НЕПРАВИЛЬНО
```

Скрипт считал, что offset 0x10 — это «размер заголовка», и начинал читать данные
с середины секции данных, получая в итого 750 отсчётов вместо 3000.

**Стало** (исправлено):
```python
wav_data_start = u32(data, 0x10)     # 12089 — начало данных
wav_data_size  = u32(data, 0x14)     # 6000
ch1_enabled    = data[0x66]          # флаг CH1
ch2_enabled    = data[0xA6]          # флаг CH2
sample_count   = u32(data, 0x12D)    # 3000 — из поля buffer_sample_count

# CH1 начинается с wav_data_start
# CH2 начинается сразу после CH1 (если активен)
```

### 3. Переименование BASE_SETTINGS_SIZE

**Было**: `BASE_SETTINGS_SIZE = 11981` с комментарием «Minimum settings header size».

Это значение — не «минимальный размер заголовка настроек», а минимальное значение
`waveform_data_start` при пустой таблице буферов (50 header + 911 base settings +
11020 screen buffer = 11981).

**Стало**: `BASE_WAV_DATA_START = 11981` с корректным комментарием.

### 4. Добавлена валидация таблицы секций

при несовпадении.

### 5. Добавлена информация о буфере экрана

соответствие размера.

## Использование dump_header.py

```bash
# Один файл:
python dump_header.py Waveform-google/1MHz-1V.wav

# Все файлы:
python dump_header.py Waveform-google/*.wav
```

Вывод включает:
- Таблицу секций с валидацией ✓
- Все известные поля настроек с hex/dec значениями
- Параметры буфера экрана (RGB565 102×54)
- Статистику данных АЦП (min/max/median на канал, V/div)
- Raw hex-дамп первых 96 байт
